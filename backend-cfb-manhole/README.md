# Backend CFB Manhole API (Go)

A Go API for leaderboard data and player persistence backed by Neon Postgres.

This service stores each player's app-level ID, current score, and selected teams in Postgres. On first startup against an empty database, it seeds the initial player/team set from the existing in-repo scoring data.

## Requirements

- Go 1.26+
- A Neon Postgres connection string exposed as `NEON_DATABASE_URL` or `DATABASE_URL`

## Run

From this folder:

```bash
export NEON_DATABASE_URL="postgres://..."
go run .
```

Server starts at `http://127.0.0.1:8000`.

## Architecture Role

The backend is responsible for:

- serving ranking data over HTTP,
- persisting player scores and player teams in Neon Postgres,
- seeding initial players and teams into a new database,
- regrouping global rankings into division-specific leaderboards.

The main code paths are:

- `main.go`: server startup and wiring.
- `handlers.go`: HTTP request handling and route matching.
- `service.go`: ranking and player service logic.
- `database.go`: Postgres schema bootstrapping, seeding, and repository queries.
- `scoring.go`: initial in-repo scoring and seed-source data.
- `responses.go`: JSON response helpers and CORS headers.

## Endpoints

- `GET /health`
  - Returns service status.
- `GET /api/rankings`
  - Returns players ordered by descending score.
- `GET /api/rankings/divisions`
  - Returns all divisions with division-specific rankings.
- `GET /api/rankings/divisions/{divisionId}`
  - Returns a single division leaderboard.
  - Example: `/api/rankings/divisions/1`
- `GET /api/players`
  - Returns all players with their app-level IDs, current scores, and teams.
- `POST /api/players`
  - Creates a player record and the player's teams.
- `GET /api/players/{playerId}`
  - Returns one player record.
- `DELETE /api/players/{playerId}`
  - Deletes one player record and its related teams.
- `PUT /api/players/{playerId}/score`
  - Overwrites the player's latest score.
- `PUT /api/players/{playerId}/teams`
  - Replaces the player's saved teams.

## Data Model

The backend persists the following tables in Postgres:

- `players`
  - `id`: app-level UUID generated by the backend
  - `name`: unique player name
  - `division`: division number
  - `current_score`: latest persisted score
- `player_teams`
  - one row per player/team selection

The legacy in-memory scoring data still exists only as a seed source for a new database.

## CORS

This API includes permissive CORS headers for frontend wiring:

- `Access-Control-Allow-Origin: *`
- `Access-Control-Allow-Methods: GET, POST, PUT, DELETE, OPTIONS`
- `Access-Control-Allow-Headers: Content-Type`
